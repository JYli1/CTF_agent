# CTF Web 题目通用起手手册

## 使用说明

- 适用场景：给定一个 Web 题目的 URL，希望从零开始做信息收集与初步分析。
- 目标：在**不依赖题解**的前提下，快速获得足够多的“靶场画像”，为后续漏洞挖掘（SQLi / XSS / RCE / 逻辑漏洞等）打基础。
- 本文分为：
  - 通用起手流程（任何 Web 题都可以套用）
  - 针对常见类型的“起手 check list”（SQLi / XSS / 文件上传 / SSRF 等）

---

## 一、总体起手流程（总览）

1. **基本连通性检查**
   - 访问题目 URL，确认站点能访问，记录状态码 / 跳转情况。对于前期信息收集，为了准确性和效率，建议使用 `curl` 或 `wget` 命令。
   - 记录：协议（HTTP/HTTPS）、端口、是否有明显的重定向或负载均衡迹象。

2. **页面结构与技术栈初步判断**
   - 查看 HTML 源码，记录框架痕迹（如 Vue/React、某 CMS、某语言特有注释）。
   - 记录响应头里的 Server / X-Powered-By / Set-Cookie / CSP 等信息。

3. **公开信息与常见敏感文件枚举**
   - 访问 `robots.txt`、`/.git`、`/backup/`、`/phpinfo.php` 等常见路径。
   - 枚举常见备份文件：
     - `index.php.bak` / `index.php~` / `.index.php.swp` / `index.php.save` 等。

4. **输入点与功能点枚举**
   - 识别页面上的所有表单、链接上的 GET 参数、JS 动态请求中的参数。
   - 记录：每个参数所在 URL、HTTP 方法、作用（登录、搜索、上传等）。

5. **会话与权限相关信息收集**
   - 观察 Cookie 名称、是否有 Session ID / JWT / 自定义 token。
   - 观察是否存在登录/注册/找回密码/后台入口等。

6. **根据功能类型选择对应“起手 check list”**
   - 搜索框 / 列表筛选：优先考虑 SQL 注入、XSS、SSTI。
   - 上传功能：文件上传绕过 / 文件包含 / RCE。
   - 跳转、URL 参数中带 URL：考虑 SSRF / Open Redirect。
   - 序列化字段 / 大块长字符串：考虑反序列化。

## 二、基础信息收集（任何题目都要做）

### 1. 访问主页并记录关键信息

- 打开题目 URL，记录：
  - HTTP 状态码（200 / 301 / 302 / 403 / 500 等）。
  - 页面主体内容（有无提示、题面描述、明显关键字）。
  - 有无“Forbidden / Not Found / Error”之类的错误信息。

- 查看源代码（Ctrl+U 或浏览器开发者工具）：
  - 注释中是否有路径、账号、调试信息。
  - JS 文件引用路径，如 `/static/js/app.js`、`/assets/...` 等。
  - 是否有明显框架痕迹（如 `<meta name="generator" ...>` 或特殊标签）。

### 2. 查看响应头与 Cookie

- 重点字段：
  - `Server` / `X-Powered-By`：泄露语言、框架、版本（如 `Apache/2.4`、`PHP/7.x` 等）。
  - `Set-Cookie`：Session 名称、`HttpOnly` / `Secure` 标志。
  - `Content-Security-Policy`：对 XSS / iframe 等有影响。
  - 任何自定义 header：`X-Frame-Options`、`X-Debug` 等。

- 目的：
  - 判断是否易受 XSS / Clickjacking / CSP 绕过等影响。
  - 判断是否可能利用某些框架已知漏洞（低版本 WordPress / ThinkPHP 等）。

### 3. 公共文件与目录探测（轻量版）

- 常规访问：
  - `/robots.txt`
  - `/sitemap.xml`
  - `/phpinfo.php`、`/info.php`
  - `/admin/`、`/login/`、`/manage/` 等常用后台路径。

- 常见备份文件：
  - `index.php.bak`
  - `index.php~`
  - `.index.php.swp`
  - `index.php.save`
  - 某些题目会故意放 `index.php.txt` 暴露源码。

> 在 CTF 场景中，这一步经常直接给出源码或 Flag 线索，是性价比极高的起手操作。

## 三、输入点收集与分类

### 1. GET / POST 参数

- 列出所有带参数的 URL：
  - 如 `?id=1`、`?page=1&sort=asc` 等。
- 对每个参数记录：
  - 参数名 / 默认值
  - 页面的功能：查看详情 / 搜索 / 过滤 / 登录等。

### 2. 表单与上传点

- 识别：
  - 登录 / 注册 / 修改密码 / 搜索 / 评论 / 留言板等表单。
  - `input type="file"` 的上传功能。
- 对上传点记录：
  - 限制类型（前端 accept / 后端返回的报错）。
  - 上传成功后文件访问路径（若能直接访问）。

### 3. Header 与 Cookie 可控性

- 尝试是否可以控制：
  - 自定义 header（`X-Forwarded-For`、`X-Real-IP` 等）。
  - Cookie 中的 user_id / role / token 等字段。
- 留意：
  - Cookie 是否是明文 JSON / base64。
  - 是否有明显的序列化数据（`O:...`、`s:...` 等 PHP 序列化格式）。

## 四、按照功能选择“起手 check list”

### 1. 有 ID / 数字参数 → SQL 注入 / 数字型漏洞 起手

**基础测试：**

- 在参数后追加：
  - `'`、`"`、`\`，观察是否报 SQL 错误。
- 经典布尔盲注差异测试：
  - `id=1 and 1=1`
  - `id=1 and 1=2`
  - 比较两次响应长度、内容差异。

**注意事项：**

- 如果是搜索框类参数，可尝试 `' or '1'='1` 等拼接。
- 对于明显使用 ORM/框架的站点，错误信息可能被吞掉，需要更精细的长度/时间对比。

### 2. 显示用户输入的页面 → XSS 起手

**基础测试：**

- 在可见输入点输入简单 payload：
  - `<script>alert(1)</script>`
  - `<img src=1 onerror=alert(1)>`
- 查看：
  - 是否被原样输出；
  - 是否被 HTML encode (`&lt;script&gt;`)；
  - 是否只在 JS 字符串 / 属性中出现。

**分场景思路：**

- HTML 上下文 → 直接标签型 XSS。
- 属性上下文 → 需要引号闭合 + 事件属性。
- JS 字符串上下文 → 需要闭合引号 + 拼接 JS 代码。

### 3. 有模板渲染痕迹 → SSTI 起手

**特征：**

- 页面上有 `{username}`、`${username}`、`{{ username }}` 等占位符。
- URL 或参数看起来会被带入模板。

**基础测试：**

- 尝试 payload（按常见模板引擎）：
  - `{{7*7}}`、`{{7*'7'}}`
  - `${7*7}`
- 如果页面回显 `49` 或异常字符串，基本可以确认 SSTI 方向。

## 五、文件上传 / 文件包含 / RCE 起手

### 1. 文件上传功能起手

**检查点：**

- 前端限制：
  - 通过 HTML `accept` 或 JS 只允许某些后缀？（可以抓包改）
- 后端限制：
  - 上传 `.php`、`.phtml`、`.phar` 时返回什么？
  - 是否支持压缩包 / 图片 exif / 双后缀（`shell.php.jpg`）？

**基础起手操作：**

- 尝试上传：
  - 图片+修改后缀（`.jpg`）；
  - 双扩展名（`shell.php.jpg`）；
  - 大小写混合后缀（`.PhP`）；
  - 特殊后缀（`.phtml`、`.phar`）。

- 观察返回：
  - 是否返回保存路径；
  - 是否能从 URL 访问到上传文件。

### 2. 文件包含 / LFI 起手

**典型特征：**

- 有参数名类似 `file` / `page` / `tpl`：
  - 如 `index.php?page=about`。

**基础测试：**

- 传入 `../etc/passwd` 等路径，看是否报错或回显文件内容。
- PHP Wrapper 起手：
  - `php://filter/read=convert.base64-encode/resource=index.php`

**注意：**

- 很多 CTF 题会把 LFI + 上传结合起来：先上传文件，再通过 LFI 包含上传的临时文件或日志。

## 六、SSRF / URL 跳转 起手

### 1. 参数中出现 URL / host

**特征：**

- 参数值类似：`url=http://xxx` / `target=http://xxx` / `redirect=...`。

**基础检查：**

- 将其改为自己的服务器地址，观察服务器是否收到请求。
- 试试内网地址：
  - `http://127.0.0.1/`
  - `http://localhost/`
  - `http://169.254.169.254/`（云环境元数据服务）

### 2. Open Redirect 起手

- 将 redirect 参数改为：
  - `https://www.google.com/`
  - 或自己控制的站点。
- 查看：
  - 是否能被重定向出去；
  - 是否存在前缀拼接（可绕过黑名单）。

## 七、序列化 / 逻辑类起手

### 1. 序列化与 Token 起手

**特征：**

- Cookie 中有长串 base64 / URL-safe 字符串。
- 参数中存在 `data=` / `payload=` 等大块数据。

**基础操作：**

- 先尝试 base64 解码，看是否是 JSON / PHP 序列化 / JWT。
- 如果是 JWT：
  - 查看 header / payload 是否可控；
  - 是否使用弱密钥 / 无签名验证。

- 如果是 PHP 序列化：
  - 字符串中有 `O:`、`s:`、`a:` 等结构；
  - 考虑 __wakeup / __destruct 链，但这是后续利用步骤，起手先确认可控性与解码结果。

### 2. 逻辑漏洞起手（没有明显技术洞时）

- 检查：
  - 是否有购物车 / 支付 / 积分 / 抽奖等业务逻辑。
- 基础操作：
  - 重复提交请求（抢购 / 抢券）；
  - 篡改价格 / 数量参数；
  - 非正常流程访问某些接口（先访问支付结果页面，再访问订单页等）。

## 八、实战时的通用“起手提问模板”

当你只拿到一个 URL，可以让 Agent 先遵循如下思路：

1. **信息收集阶段**
   - 访问主页、记录响应头和 Cookie。
   - 枚举常见敏感文件和备份文件。
   - 列出所有 GET / POST 参数和表单。

2. **基于功能的初步分类**
   - 按输入点类型（搜索 / 登录 / 上传 / 跳转）挑选对应的“起手 check list”。

3. **第一轮验证**
   - 对每个疑似点，先做“轻量无损测试”：
     - SQLi：加入简单的 `'` / `and 1=2` 差分。
     - XSS：无害的 `<img src=x onerror=alert(1)>`，只在题目环境使用。
     - SSRF：请求自己的回调地址，确认出网能力。
     - 文件上传：尝试上传正常图片 + 伪后缀文件，对比行为差异。

4. **记录与复盘**
   - 记录每一步的请求与响应特征：
     - 哪些参数有明显异常；
     - 哪些路径可以访问到源码/备份；
     - 哪些操作会引起 500/报错页面。

这些信息会成为后续深挖漏洞（注入、RCE、提权）的基础。
